# PrototypeDemo — Prototype (Клонирование профилей пользователей)

## Что такое Prototype и почему он здесь
**Prototype (Прототип)** — порождающий паттерн, который позволяет создавать новые объекты путём клонирования существующих. В данном проекте он используется для **клонирования профилей пользователей**, потому что:

- **Есть готовые шаблонные профили** (например, профиль для маркетинга, профиль для поддержки). Вместо того чтобы каждый раз создавать их с нуля, мы храним их как «прототипы».
- **Нужно быстро создавать новые профили** на основе этих шаблонов, меняя только часть данных (логин, язык, права). Прототип позволяет сделать это одной операцией `Clone()`.
- **Сложная структура объекта**: профиль содержит вложенные объекты (`Preferences`, `NotificationSettings`) и коллекции (`Permissions`). Без паттерна пришлось бы вручную копировать все поля, что легко привести к ошибкам.

## Где именно используется Prototype в коде

### 1. Прототипы (шаблонные объекты)
```csharp
var marketingTemplate = new UserProfile(...);
var supportTemplate = new UserProfile(...);
```
Эти объекты — «эталоны». Они не используются напрямую, а служат основой для клонирования.

### 2. Операция клонирования
Класс `UserProfile` реализует два способа клонирования:
- `ShallowClone()` — поверхностное копирование (использует `MemberwiseClone()`). Вложенные объекты и коллекции остаются общими между прототипом и клоном.
- `DeepClone()` — глубокое копирование. Создаются независимые копии всех вложенных объектов и коллекций.

### 3. Каталог прототипов как фабрика
```csharp
var profileCatalog = new Dictionary<string, UserProfile>
{
    ["marketing"] = marketingTemplate.DeepClone(),
    ["support"] = supportTemplate.DeepClone(),
};
```
Здесь прототипы выступают в роли «мини-фабрики»: по ключу получаем готовый шаблон, клонируем его и кастомизируем.

## Зачем нужны оба типа клонирования
- **Shallow clone** демонстрирует проблему: при изменении вложенного объекта в клоне меняется и прототип. Это важно понимать, чтобы избежать побочных эффектов.
- **Deep clone** решает эту проблему: изменения в клоне не затрагивают прототип, что делает его безопасным для независимых экземпляров.

## Почему Prototype лучше альтернатив
- **Без конструкторов с десятками параметров**: не нужно передавать все поля при создании нового профиля.
- **Снижение ошибок**: автоматическое копирование полей через `Clone()` уменьшает риск что-то забыть.
- **Производительность**: если создание объекта «дорого» (например, загрузка данных из БDI), клонирование может быть быстрее.

- `PrototypeDemo/NotificationSettings.cs` — настройки уведомлений
